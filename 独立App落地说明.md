# 专属日历看板独立 App 落地说明

## 当前版本定位

当前示例是一个手机优先的轻量 Web/PWA 原型，适合验证界面结构、四库关系、今日计划展示、完成打勾、复盘摘要和双人共享逻辑。

它还不是完整独立 App，暂未接入真实账号、云同步、系统日历权限和桌面小组件。

## 推荐技术路线

### 第一阶段：轻量桌面入口

- 使用当前 PWA 原型继续完善界面。
- 安卓手机可通过浏览器“添加到桌面”形成近似 App 的入口。
- 强提醒先通过生成 `.ics` 日历文件导入系统日历实现。
- 数据先放在本地示例或本地 JSON，便于快速调整字段。

### 第二阶段：多人同步

推荐使用 `Supabase` 或 `Firebase` 作为云端数据源，而不是飞书多维表格。

原因：

- 支持账号登录。
- 支持实时同步。
- 支持权限控制。
- 适合后续独立 App。
- 可以同时支持你和对象的双人空间。

建议数据结构：

| 表 | 作用 |
|---|---|
| `users` | 你和对象的账号资料 |
| `couple_spaces` | 双人共享空间 |
| `calendar_items` | 总日历库 |
| `tasks` | 任务库 |
| `knowledge_notes` | 知识库 |
| `reviews` | 复盘库 |
| `permissions` | 共享权限 |
| `reminder_jobs` | 提醒任务 |

### 第三阶段：独立 App

推荐优先考虑 `Flutter` 或 `React Native`。

如果目标是安卓优先，尤其是 OPPO 和小米：

- `Flutter` 更适合做稳定的跨端 App 和桌面小组件。
- `React Native` 更适合前端 Web 逻辑迁移。
- 如果后续只做安卓，也可以用原生 Kotlin。

## 安卓强提醒方案

### 推荐优先级

1. 系统日历事件：适合考试、就医、定投、重要计划。
2. App 本地通知：适合普通任务、学习提醒、运动打卡。
3. 前台常驻/厂商白名单：只用于极强提醒，不建议一开始做。

### OPPO / 小米注意点

- 需要用户允许通知权限。
- 需要允许日历权限。
- 如果做 App 通知，需要提示用户关闭电池优化或允许后台运行。
- 小米和 OPPO 对后台通知限制较多，所以重要事项建议写入系统日历。

## 双人共享设计

建议默认采用“可见但不可改”的共享方式：

| 内容 | 默认权限 |
|---|---|
| 今日完成率 | 对象可见 |
| 今日任务标题 | 对象可见 |
| 学习知识库 | 可按模块共享 |
| 健康记录 | 默认隐藏，手动共享 |
| 投资金额 | 默认隐藏 |
| 投资提醒 | 可共享提醒，不共享金额 |
| 灵感收集 | 默认私密，部分可共享 |
| 对象知识库 | 需要对象授权后可见 |

## 首页展示原则

按照你的偏好，首页只展示今天：

- 今天的关键事项。
- 今日待办数量。
- 今日高优先级。
- 今日完成率。
- 今天是否有强提醒。
- 对象可见的今日进度摘要。

本周、逾期、长期计划不放首页，避免压力过大；这些内容进入四库详情页或复盘页。

## 下一步建议

1. 先确认四个库的字段是否够用。
2. 再确认双人共享权限是否符合你和对象的边界。
3. 然后把原型升级为可保存数据的版本。
4. 最后再接入云端同步和安卓系统日历。
